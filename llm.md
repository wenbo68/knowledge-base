# pre-training vs fine-tuning vs evaluation
1. pre-training: model with random weights -> model with weights that allows it to predict next words nicely
    - dataset: just rows of texts (1 column)
2. fine-tuning: adjust model behavior/style
    - creating lora/qlora adapters/matrices
        - if your ai app have different domains...
            - you can use 1 base inference model + 1 adapter for each domain to fine tune + 1 rag for each domain (to give domain knowledge)
        - can merge the adapter weights to the base model: faster inference speed (but unlikely if you have multiple domains)
        - OR keep adapter and base model separate (only use together at run time)
    - dataset
        - mostly generated by AI based on a small number of human-written entries
        - 2 main columns
            - user input
            - expected output: the perfect answer you want the LLM to generate
            - (if rag) expected retrieved nodes: the list of nodes/chunks that should be retrieved
3. evaluation: score the model after it is freshy made or after you changed sth
    - 2026 tools: DeepEval (the orchestrator) + Ragas (the grading rubrics)
        1. DeepEval: The "Teacher"
            - automates the testing loop (runs your eval dataset against your app)
            - spins up a "Judge LLM" (like GPT-4o) to grade your app's responses
            - integrates all this testing into your CI/CD pipeline
        2. Metrics & Ragas: The "Grading Rubrics"
            - a "metric" is just a specific grading formula/prompt you give to the Judge LLM.
            - DeepEval has its own built-in metrics (e.g., "Summarization Score", "Toxicity Score").
            - Ragas is a famous set of RAG-specific metrics (formulas for measuring "Context Precision", "Answer Relevance", etc.).
            - You literally tell DeepEval: "Hey, use the Ragas metrics when you spin up your Judge LLM."
    - dataset
        - 3 main parts
            - the good (regression test): queries the model must get right every time
            - the bad: queries the model previously failed
            - the hard (stress test): tricky queries that test reasoning limits
        - 2 main columns (same as fine-tuning): user input + expected inference output + (if rag) expected retrieved nodes

# debugging llm
- python code might be right, but llm behavior or thinking process might be wrong (or it might just hallucinate)
- steps:
    1. you find llm doing wrong things (give bad answer, wrong thought process, etc.) in langsmith to a certain user query
    2. add the query to your evaluation dataset (can be done/created within langsmith via "add to dataset")
    3. first check RAG if you have one
        - if retrieved nodes are wrong/missing, you need better parsing/chunking or better embedding models
            - but switching embedding models is very expensive as it requires re-indexing the entire vector db
    4. switch to best inference model first and evaluate using the same query
        - if better inference fails too: your system prompt is garbage OR the task is impossible
        - if better inference works...
            - you can either just use the better model OR route easy queries to simple inference and hard queries to more costly inference
            - OR proceed with the following steps
    5. system prompt engineering: industry system prompts can be 2000+ words
        - manual few-shotting: add a generalized version of the query and the correct behavior or thought process to the system prompt
            - only add 0-5 examples to system prompt
        - dynamic few-shotting: let rag retrieve few-shot examples and add to system prompt
            - 5-50 examples in vector database
    6. if still fails OR if too many few-shots OR if latency issues from few-shotting...
        - if it's a knowledge problem: add rag or internet search
        - if it's a behavior/style problem: fine tune the inference model
